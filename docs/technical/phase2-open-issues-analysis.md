## 1. 언어 지원 확장 시점

**현 상태(mvp-spec §1.1)**: Python, JavaScript/TypeScript 우선 지원. 그 외 언어는 "Best-effort"로 표기.

### 선택지

1. **전면 Best-effort 허용**: Go/Rust/Ruby 등 비지원 언어도 동일 UI로 연결 가능
   - 장점: 얼리 어독터 pool 최대화, 커뮤니티 요구 학습 기회
   - 단점: 저품질 PR로 초기 신뢰도 훼손, CS 부담 증가
2. **명시적 차단**: 연결 시점에 언어 감지 후 비지원이면 차단 + 대기리스트
   - 장점: 품질 일관성, CS 부담 최소화
   - 단점: 다언어 1인 개발자(Go backend + Next.js frontend 등) 유입 장벽
3. **Best-effort + `beta` 배지(권장)**: 비지원 언어 레포도 연결 허용하되 대시보드에 "beta 지원" 배지, PR 본문에 "beta 품질" 경고 문구
   - 장점: 유입 확대 + 기대치 관리 병행
   - 단점: 배지/문구 유지 비용

### 권장안과 이유
**선택지 3(Best-effort + beta 배지)**. 얼리 어독터 단계(10~50명)에서는 실사용 데이터가 곧 자산. 품질 리스크는 UX 문구(배지·PR 경고)로 완화 가능. 6주 뒤 머지율·만족도 데이터로 재평가.

### 결정 시 DECISIONS.md 기록 예시
> "2026-04-24: 언어 지원은 Python·JS/TS P0, Go/Rust/Ruby는 beta Best-effort. 대시보드 배지 + PR 본문 경고 문구 적용. 재평가 시점: 얼리 어독터 20명 돌파 또는 MVP+6주."

### 후속 구현 영향
- 레포 분석 파이프라인(F-02): 언어 감지 로직 + `language_support_tier` 필드 추가
- 이메일 리포트(F-04): beta 레포는 리포트에 "beta" prefix

## 2. 500MB 이상 대형 레포 분석 전략

**현 상태(mvp-spec §F-02)**: `git clone --depth=50` 얕은 클론만 명시. 레포 크기 상한 정의 없음.

### 데이터 포인트(추정)
- MVP 타깃 인디 해커 레포 p90 크기: 100MB 이하(React/Next.js + 에셋 수백MB 레포 드물게 포함).
- 500MB 초과: mono-repo, 게임 에셋, ML 모델 포함 레포. MVP 타깃에서 전체의 5% 이하로 추정.

### 선택지

1. **얕은 클론 + partial clone**: `git clone --depth=50 --filter=blob:limit=10M` 로 blob 크기 제한
   - 장점: 기존 흐름 유지, 구현 단순
   - 단점: 10MB 초과 asset은 fetch 불가 → ML/게임 레포 분석 불가
2. **파일 샘플링**: README, 매니페스트(`package.json` 등), 최상위 설정, 최근 수정 파일 상위 N개만 fetch (GitHub API `contents` 엔드포인트)
   - 장점: 디스크/네트워크 부담 최소
   - 단점: 깊이 있는 컨텍스트 부족 → 리팩터링/테스트 보강 품질 저하
3. **거절**: 2GB+ 레포 연결 차단, 2GB 이하는 전체 클론
   - 장점: 보수적, 운영 리스크 최소
   - 단점: mono-repo 기업 CTO 유입 어려움(Post-MVP 타깃이지만)
4. **하이브리드(권장)**: 500MB 이하 → (1) 얕은 클론 · 500MB~2GB → (2) 샘플링 · 2GB+ → (3) 거절

### 권장안과 이유
**선택지 4(하이브리드)**. MVP 타깃 95%는 500MB 이하로 기존 흐름 유지, 5% 대형 레포도 샘플링으로 "부분 가치" 제공, 극단치(2GB+)는 운영 비용·런타임 상한 리스크로 거절. 샘플링 상세 규칙은 별도 설계 문서 필요.

### 별도 설계 문서 필요 여부
**필요**. `docs/technical/large-repo-strategy.md` 를 Post-MVP 착수 전 추가. 샘플링 규칙, 토큰 예산, 유저 UX(거절 시 대기리스트) 정의.

### 후속 구현 영향
- F-01 온보딩: 레포 연결 직후 크기 측정 → 2GB+ 거절 모달
- F-02 분석 파이프라인: `analysis_mode ∈ { full, sampled }` 분기
- 비용: 2GB+ 거절로 Railway 디스크 예산 예측 가능

## 3. "에이전트가 고장낸 PR" 처리

**현 상태(mvp-spec §F-03)**: "7일간 머지되지 않은 PR은 자동 `stale` 코멘트 1회만" 명시. 그 이후 처리 미정.

### 선택지

1. **7일 후 자동 close**: 에이전트가 스스로 PR 닫음
   - 장점: PR 리스트 청소, 사용자 주의 분산 방지
   - 단점: 사용자 피드백 루프 단절(왜 머지 안 했는지 에이전트가 배울 수 없음)
2. **7일 후 `stale` 라벨 + 알림만(권장)**: 라벨 부여 + 이메일 리마인드 1회, close는 사용자 결정
   - 장점: 사용자 주권 유지, 라벨로 검색 필터링 가능
   - 단점: 장기적으로 open PR 누적
3. **7일 후 사용자 선택 이메일**: "close? keep open? give feedback?" 버튼 포함 메일
   - 장점: 명시적 피드백 확보 기회
   - 단점: 추가 인프라(mailto 토큰 서명 등) 필요, 응답률 낮을 위험

### 권장안과 이유
**선택지 2 + 30일 cap**. 7일 → `stale` 라벨 + 이메일 리마인드(1회), 30일 → 에이전트가 "이 PR을 닫습니다" 코멘트와 함께 자동 close. 사용자는 언제든 close 또는 라벨 제거 가능. 사용자 주권을 우선하되 장기 누적을 방지.

### 후속 구현 영향
- F-03: `pr_lifecycle` cron job(일 1회, 라벨 부여 + 30일 close) 추가
- 이메일 리포트: "stale 3건" 요약 섹션 추가(선택적)

## 4. Free 플랜 코드 전송 범위

**현 상태(mvp-spec §6 NFR)**: "레포 코드 원본 저장 금지, Claude API 호출 시 캐시 비활성화".

### 핵심 혼선
NFR은 "저장 금지"를 명시하나, 분석 품질을 위해 파일 내용을 **일회성으로 Claude API에 전송**하는 행위가 "저장"에 해당하는지 불명확.

### 정의 재정리(권장)
- **"저장(Storage)"**: ShipCrew 서버 디스크/DB/Redis 등 영속 계층에 코드 원본을 쓰는 행위 → **금지**
- **"전송(Transit)"**: Claude API 호출 시 프롬프트에 파일 내용 포함 → **허용**, 단 `cache_control` 비활성화
- **"임시 작업 디렉토리(Ephemeral workspace)"**: Celery 워커의 `/tmp` 클론 → **허용**, 작업 종료 후 즉시 삭제(mvp-spec §F-02 수용 기준에 이미 명시)

### 선택지(전송 허용 수준)

1. **메타데이터만 전송**: 파일 목록, 매니페스트, README, TODO 주석만
   - 장점: 프라이버시 최강, 프롬프트 토큰 최소
   - 단점: 분석 깊이 부족 → 리팩터링/테스트 보강 제안 품질 급락
2. **시그니처·docstring까지**: (1) + 함수 시그니처, 클래스 정의, docstring
   - 장점: 구조 컨텍스트 확보
   - 단점: 구현체 없이는 오판 쉬움
3. **전체 파일 포함(권장)**: 분석 대상 파일 최대 20개의 풀 내용 전송, 캐시 금지
   - 장점: 최고 품질
   - 단점: 프라이버시 FAQ에서 "저장" vs "전송" 구분 필수

### 권장안과 이유
**선택지 3**. 얼리 어독터 단계에서 PR 품질은 최우선 리텐션 요인. Anthropic API 계약상 "학습 미사용"은 이미 보안 FAQ에서 명시(2026-04-13 DECISIONS). 추가로 `cache_control=disabled` + "서버 캐시 금지" 명시하면 프라이버시 원칙 유지.

### 반드시 병행할 작업
- **프라이버시 FAQ 업데이트**: "저장"/"전송"/"임시 작업 디렉토리" 3개 개념 분리 설명
- **Pro 플랜 추가 옵션(Post-MVP)**: "코드 전송 금지 모드"(메타데이터만) 토글 → 고보안 고객 대응

## 5. 이메일 공급자 선택

**현 상태(mvp-spec §F-04)**: "SendGrid or Resend" 병기, 미결.

### 공급자 비교

| 항목 | Resend | SendGrid | Postmark |
| --- | --- | --- | --- |
| 개발자 경험 | ★★★★★ (React Email) | ★★★ (템플릿 HTML 수기) | ★★★★ |
| 도달률(업계 공개치) | 미공개 | 95%+ | 99%+(트랜잭셔널 특화) |
| 무료 티어 | 월 3,000통 | 월 100통 | 월 100통 |
| 유료 시작가 | $20/월(50k) | $19.95/월(40k) | $15/월(10k) |
| MVP 트래픽(100명·일1통) | 3,000통 → **무료 안쪽** | 유료 필요 | 유료 필요 |
| 한국 도달률 | 불명 | 평균 | 우수 |

### 선택지

1. **Resend(권장)**: 개발 속도 + 무료 티어 여유 + React Email 재사용 가능
2. **SendGrid**: 엔터프라이즈 도달률 대시보드 필요할 때
3. **Postmark**: 도달률 0.5~1% 차이가 중요할 때

### 권장안과 이유
**선택지 1(Resend) + 3개월 후 재평가 A/B**. 이유: 개발 속도 > 도달률 미세 차이(MVP 단계 유저 규모). React Email 컴포넌트로 템플릿 작성 → 대시보드 디자인 재사용 가능. 3개월 후 실측 오픈율/스팸율로 공급자 A/B.

### 비용 시뮬레이션
- Month 1(waitlist 50명): 50통 × 30일 = 1,500통 → 무료
- Month 3(Pro 30명 + waitlist 200명): 230통 × 30일 ≈ 7,000통 → Resend $20/월 플랜
- Month 6(Pro 100명): 약 30,000통 → 여전히 $20/월 플랜 내

## 6. Stripe 세금/인보이스

**현 상태(mvp-spec §F-06)**: "부가세/세금계산서는 MVP 제외 — UI에 '영수증은 Stripe 고객 포털' 문구".

### 법적·운영 배경
- 한국 거주 사업자가 한국 거주자에게 전자적 용역을 공급 시 부가세 10% 과세 대상.
- 개인사업자 고객의 세금계산서 요청은 B2B SaaS에서 빈번. 미대응 시 구매 포기 사유.
- Stripe Tax는 전 세계 VAT/GST 자동 계산 지원. 인보이스는 Stripe Invoicing 또는 한국 국세청 연동 외부 서비스 필요.

### 선택지

1. **전면 Post-MVP 보류(권장)**: `Billing` 페이지에 "세금계산서 필요 시 billing@shipcrew.dev 개별 문의" 문구. B2C 개인 개발자 위주라는 MVP 타깃 전제에 부합.
2. **Stripe Tax만 활성화**: 자동 VAT 계산, 한국/EU/일본 부가세 반영
   - 장점: Global 사용자 컴플라이언스 확보
   - 단점: 한국 세금계산서는 여전히 미대응
3. **Stripe Tax + Stripe Invoicing**: B2B 인보이스 발급, 사업자 정보 수집
   - 단점: MVP 8주 일정 대비 오버스펙

### 권장안과 이유
**선택지 1**. MVP 타깃인 인디 해커/1인 개발자는 대부분 B2C 개인 결제. 최초 Pro 고객 10명 중 세금계산서 요청이 생기면 수동 대응(한국 홈택스 공급자 직접 발행) 후 P1으로 승격. 무리한 자동화가 8주 로드맵 리스크.

### 대응 템플릿(권장 문구)
- Billing 페이지: "세금계산서는 구독 후 이메일(billing@shipcrew.dev)로 요청하시면 영업일 3일 이내 발행해드립니다."
- 한국어 이메일 자동응답 템플릿 1종 준비.
